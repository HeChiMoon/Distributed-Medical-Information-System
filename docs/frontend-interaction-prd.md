# DMIS 前端真实业务交互与演示交付版 PRD

## 1. 背景与问题

DMIS 已完成 Spring Boot 3 微服务后端、API 网关、Nacos 注册与配置中心、Redis 缓存、Feign 远程调用、Swagger/OpenAPI 文档和单元测试。当前 Vue3 前端仍以静态演示驾驶舱为主，无法直接完成患者、预约、病历、医嘱、药房、账单、系统管理等真实业务操作，也无法直观看到登录认证、网关访问、缓存、服务间调用的前端交互闭环。

本阶段目标是把前端升级为可演示、可操作、可验收的业务工作台：所有核心模块通过 API 网关访问真实后端接口，用户可以在页面中完成表单录入、列表查询、状态操作和演示链路验证。

当前交付还要求形成可录制的 demo 视频材料。视频中需要解释系统采用的技术栈、项目架构、已实现功能、日常启动方式和核心分布式能力，因此 README、架构文档、启动脚本和视频脚本需要与真实代码保持一致。

## 2. 目标用户

- 演示讲解者：需要在答辩或验收现场快速展示系统能力。
- 医院信息科管理员：关注患者、排班、字典、日志、服务运行状态。
- 门诊医生：关注患者信息、预约、电子病历、医嘱。
- 药房人员：关注药品维护、库存入库、发药记录。
- 收费人员：关注账单创建、明细、支付状态。

## 3. 当前状态

- 后端服务：`auth-service`、`patient-service`、`appointment-service`、`medical-record-service`、`pharmacy-service`、`billing-service`、`admin-service`、`gateway-service`。
- 统一访问入口：前端请求 `/api/**`，由 Vite 代理到 `http://localhost:9527` API 网关。
- 登录认证：`POST /api/auth/login` 返回 `data.accessToken`，业务接口通过 `Authorization: Bearer <token>` 访问。
- 已知约束：本阶段不新增后端接口，不引入复杂前端路由与状态库，优先用单页工作台完成演示闭环。

## 5. 用户流程

### 5.0 日常启动流程

1. 开发者进入项目根目录。
2. 执行 `.\scripts\start-dev.ps1`。
3. 脚本启动 MySQL、Redis、Nacos，导入 Nacos 配置，打包并启动后端服务，启动 Vue 前端。
4. 打开 `http://localhost:5173` 使用 `admin / admin123` 登录。
5. 开发或演示结束后执行 `.\scripts\stop-dev.ps1` 关闭前端、后端和 Docker 基础设施。

### 5.1 登录流程

1. 用户打开前端首页，首先进入独立登录主页。
2. 输入用户名和密码，默认演示账号为 `admin / admin123`。
3. 前端调用 `/api/auth/login`。
4. 成功后保存 `accessToken` 到 `localStorage`，进入业务工作台。
5. 登录成功后进入工作台主页，后续所有业务请求自动附加 Bearer Token。
6. 点击退出后清除 Token 和当前会话状态，并返回登录主页。

### 5.2 患者管理流程

1. 用户进入“患者管理”模块。
2. 页面加载患者列表。
3. 用户填写患者姓名、性别、生日、证件号、电话、地址、血型、过敏史和病史。
4. 点击保存后调用新增或修改接口。
5. 删除操作调用逻辑删除接口并刷新列表。
6. 点击缓存演示时调用 `/api/patients/{id}/cache-demo` 展示 Redis key、TTL 与 cache-aside 策略。

### 5.3 预约排班流程

1. 创建科室。
2. 创建医生并关联科室 ID。
3. 创建医生排班并设置日期、时段和号源数。
4. 创建预约并关联患者、医生、科室和排班。
5. 可对预约执行取消操作。

### 5.4 病历医嘱流程

1. 创建电子病历，关联患者和医生，可选关联预约。
2. 查询病历列表，按患者 ID 过滤。
3. 修改病历诊断、主诉、现病史、体格检查、建议和状态。
4. 创建医嘱，关联病历。
5. 停止医嘱时调用 `/api/orders/{id}/stop`。

### 5.5 药房流程

1. 创建药品基础信息。
2. 通过入库表单录入批号、数量、生产日期、有效期和库位。
3. 按药品 ID 查询库存。
4. 通过发药表单关联患者、医嘱和药品，生成发药记录。

### 5.6 账单流程

1. 创建账单，录入患者、关联预约/病历和至少一条明细。
2. 查询账单列表。
3. 查看账单明细与支付记录。
4. 输入支付方式和金额后提交支付。
5. 对未支付账单可执行取消。

### 5.7 检索中心流程

1. 输入患者姓名或药品关键词，查询患者档案和药品目录。
2. 输入患者 ID，聚合查询该患者的预约、病历、医嘱、账单和发药记录。
3. 输入日志模块名，查询后台操作日志。
4. 页面按结果类型分区展示，便于演示“搜索/追踪一名患者相关业务数据”的流程。

## 6. UI 状态

- 空状态：列表为空时显示“暂无数据，先创建一条演示记录”。
- 加载状态：接口请求中禁用提交按钮并显示加载文案。
- 成功状态：保存、删除、支付、取消等操作成功后展示成功提示并刷新列表。
- 错误状态：展示后端返回 `message` 或 HTTP 错误信息。
- 未认证状态：没有 Token 时只显示独立登录主页、系统能力摘要和登录表单。
- 检索状态：检索中心按患者、药品、预约、病历、医嘱、账单、发药、日志分区显示结果。

## 7. 接口约束

- 前端统一调用 `/api/**`，不直接请求服务端口。
- 所有业务接口必须携带 `Authorization: Bearer <token>`。
- 响应结构遵循 `ApiResponse<T>`：`code` 为 `0` 表示成功，真实业务数据在 `data`。
- 分页响应结构为 `PageResponse<T>`：`records`、`total`、`page`、`size`。

## 9. 风险与处理

- 风险：演示数据需要按依赖顺序创建。处理：界面提供“建议流程”和 ID 提示，保留手工填写 ID 的透明性。
- 风险：后端暂未提供全部下拉联动接口。处理：前端先以列表卡片展示已创建 ID，表单可直接填 ID。
- 风险：部分业务错误被网关统一包装。处理：前端显示可读错误，并保留集成验证面板辅助定位。
- 风险：生产级权限和审计不足。处理：本阶段只用于课程/验收演示，权限细化列入后续工作。
